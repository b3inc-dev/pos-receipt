# POS Receipt 進行整理：要件・進捗・テスト検証

**目的**: 現状の実装を要件書と照らし合わせ、進捗を把握し、テスト・検証の順番と方法を整理する。

---

## 1. 要件の整理（抜粋）

### 1.1 アプリの提供価値（要件書 §1）

| 機能 | 概要 |
|------|------|
| 精算レシート | 商業施設提出用の精算レシート出力（CloudPRNT / 注文経由の両対応） |
| 特殊返金・商品券調整 | 返金手段変更・商品券釣有り差額の後処理 |
| 領収書 | 編集可能な領収書の発行・再発行 |
| 売上サマリー | 全店舗横断の売上・予算・入店数（プロプラン） |

### 1.2 POS タイル構成（要件書 §2）

1. **精算** — 今日/指定日精算、点検レシート、再印字
2. **特殊返金・商品券調整** — 取引選択 → イベント登録
3. **領収書** — 取引選択 → 発行/再発行
4. **売上サマリー** — 日付/期間指定、入店数報告（設定ON時のみ）

### 1.3 導線（要件書 §5）

- **タイル起動**: POS ホーム → タイルタップ → 取引一覧（検索・絞り込み）→ 選択 → 処理
- **取引詳細起動**: 注文詳細画面のアクションメニューから「領収書」「特殊返金」を直接起動（現在の取引が対象）

### 1.4 領収書まわり（要件書 §8, §23.3）

- テンプレートは 1 つ固定、管理画面で編集可能
- 編集項目: ロゴ、発行者情報、宛名・但し書き、日付・金額・注文番号表示など
- 発行・再発行・発行履歴保存
- プレビュー → 確認 → 発行の流れ

### 1.5 管理画面（要件書 §10, §24）

- 一般設定、ロケーション設定、精算設定、支払方法マスタ、商品券設定、特殊返金設定、**領収書設定**、ポイント/会員施策、売上サマリー設定、予算設定、印字設定
- 領収書テンプレート: 会社情報・表示ON/OFF・プレビュー付き編集

### 1.6 公開用・自社用（要件書 §0.1）

- 同一コードベースで `shopify.app.public.toml`（公開用）と `shopify.app.toml`（自社用）の 2 本立て
- `APP_MODE`（POS 拡張のベースURL）、`APP_DISTRIBUTION`（バックエンドのプラン制御）で切り替え

### 2.5 既知の制約・残タスク（要件書 §30.3, §30.4 ベース）

- **管理画面UI**: `docs/ADMIN_UI_DESIGN_RULES.md` および POS Stock 基準との揃え（コンテンツ幅・強調カード・警告バナー色など）は一部未対応の可能性あり（`docs/ADMIN_UI_DIFF_POS_STOCK_VS_POS_RECEIPT.md` 参照）。
- **売上サマリー**: タイムゾーン（JST/UTC）による日付境界は MVP 許容。キャッシュ更新ジョブの周期は要確認。
- **テスト**: 単体テスト・E2E の仕組みは未整備（vitest/jest/cypress 等の設定は未検出）。

## 3. テスト・検証の順番と方法

### 3.1 推奨する検証の流れ（上から順に実施しやすい）

1. **環境・認証の確認**
2. **管理画面の表示・遷移・保存**
3. **領収書まわり（管理画面 ⇔ POS）**
4. **精算まわり（プレビュー・作成・印字分岐）**
5. **特殊返金・商品券（登録・精算反映）**
6. **売上サマリー・予算・入店数**
7. **プラン・課金（スタンダード/プロ・自社用）**
8. **公開用・自社用デプロイの切り替え**

### 3.2 1. 環境・認証の確認

**目的**: アプリが開き、API が叩ける状態になっているか確認する。

| 項目 | 方法 |
|------|------|
| 開発ストアにインストール | パートナー → アプリ → 開発ストアにインストール |
| 管理画面が開く | 管理画面 → アプリ → POS Receipt を開き、ホームが表示されること |
| セッション・API | ホームや設定を開いたときに 401/500 が出ないこと。診断ページ `/app/diagnostics` でセッション・DB 状態を確認 |
| 自社用時 | `env-check`（`/env-check`）で `APP_DISTRIBUTION` や API Key プレフィックスが想定どおりか確認（`DEPLOY_PUBLIC_AND_INHOUSE.md` 参照） |

### 3.3 2. 管理画面の表示・遷移・保存

**目的**: 全画面が開け、主要な設定が保存できることを確認する。

| 項目 | 方法 |
|------|------|
| ナビ表示 | AppNavBar のリンクがすべて表示され、クリックで該当ページに遷移すること |
| 設定保存 | `/app/settings` でロケーション別の ON/OFF や印字方式を変更し、再読み込みで反映されていること |
| 領収書テンプレート | `/app/receipt-template` で会社名・住所・表示項目を変更し保存 → プレビューに反映されること |
| 他設定ページ | 精算設定・支払方法マスタ・印字設定・予算設定など、少なくとも「開いて保存できる」まで確認 |

**受け入れの目安**: 要件書 §27 Epic A に近い状態（管理画面が一通り開き、設定が保存できる）。

### 3.4 3. 領収書まわり（管理画面 ⇔ POS）

**目的**: 領収書の「テンプレート編集 → プレビュー → 発行・履歴」が一連で動くことを確認する。

| 項目 | 方法 |
|------|------|
| 管理画面テンプレート | 会社名・但し書きデフォルト・表示ON/OFFを変更 → 保存 → プレビューに反映 |
| POS タイル「領収書」 | タイルタップ → 取引検索 → 1件選択 → 宛名・但し書き入力 → プレビュー → 発行 |
| 取引詳細から起動 | 注文詳細のアクションメニューから「領収書」→ その取引で宛名入力 → プレビュー → 発行 |
| 再発行 | 同一取引で再度「領収書」を開き、再発行できること |
| 履歴 | 管理画面の「領収書履歴」で発行・再発行が一覧できること（要件書 §8.5, §23.3） |

**受け入れの目安**: 要件書 §27 Epic E（領収書プレビュー・発行/再発行・管理画面編集の反映）。

### 3.5 4. 精算まわり（プレビュー・作成・印字分岐）

**目的**: 精算プレビュー・作成・印字方式（CloudPRNT / 注文経由）が要件どおり動くか確認する。

| 項目 | 方法 |
|------|------|
| プレビュー | 精算タイル → 対象日・ロケーション → プレビュー表示（総売上・内訳・支払方法別など） |
| CloudPRNT 設定時 | 精算実行で精算注文を作らず直接印字（または印字ペイロード取得）になること |
| 注文経由設定時 | 精算実行で精算注文が作成され、既存の注文レシート導線で印字できること |
| 点検レシート | 点検レシート発行で 0 円または点検用の精算が作成されること |
| 再集計・再印字 | 既存精算の再集計・再印字ができること（UI がある場合） |

**受け入れの目安**: 要件書 §27 Epic C（精算プレビュー・印字方式分岐・点検レシート）。

### 3.6 5. 特殊返金・商品券（登録・精算反映）

**目的**: イベント登録と精算への反映を確認する。

| 項目 | 方法 |
|------|------|
| タイルから | 特殊返金タイル → 取引検索 → 選択 → 特殊返金 or 商品券調整の種別で登録 |
| 取引詳細から | 注文詳細のアクションから「特殊返金・商品券」→ その取引で登録 |
| 精算反映 | 該当取引を含む日の精算プレビューに、登録したイベントが反映されていること |
| void | イベントを void にすると、精算から外れること（実装済みの場合） |

**受け入れの目安**: 要件書 §27 Epic D（イベント保存・精算反映）。

### 3.7 6. 売上サマリー・予算・入店数

**目的**: 日付/期間指定のサマリー、予算、入店数報告（設定ON時）を確認する。

| 項目 | 方法 |
|------|------|
| 日付指定 | 売上サマリータイルで日付を指定し、ロケーション別の実績・予算等が表示されること |
| 期間指定 | 期間を指定して同様に表示されること |
| 予算 | 管理画面で予算を登録・編集し、サマリーに反映されること。CSV 取込があれば実行可能であること |
| 入店数 | 管理画面で「入店数報告」が ON のロケーションのみ、サマリー画面に入店数入力が表示され、保存で反映されること |

**受け入れの目安**: 要件書 §27 Epic F（日付/期間サマリー・予算・入店数ON時のみ）。

### 3.8 7. プラン・課金（スタンダード/プロ・自社用）

**目的**: スタンダードでは売上サマリーが使えず、プロでは使えること。自社用では全機能使えることを確認する。

| 項目 | 方法 |
|------|------|
| スタンダード | 売上サマリータイルまたは API が制限され、プロ案内が出ること（実装方針に依存） |
| プロ | 売上サマリーが利用可能であること |
| 自社用 | `APP_DISTRIBUTION=inhouse` のとき、プラン制限なしで全機能使えること |

**受け入れの目安**: 要件書 §27 Epic G。

### 3.9 8. 公開用・自社用デプロイの切り替え

**目的**: 同一コードベースで公開用と自社用の両方が正しく動くことを確認する。

| 項目 | 方法 |
|------|------|
| 公開用 | `APP_MODE=public` でビルドし、`shopify.app.public.toml` でデプロイ。管理画面・POS が公開用 URL で動作すること |
| 自社用 | `APP_MODE=inhouse` でビルドし、`shopify.app.toml` でデプロイ。管理画面・POS が自社用 URL で動作すること |
| 環境変数 | Render 等で公開用/自社用で異なる `SHOPIFY_APP_URL`・`SHOPIFY_API_KEY`・`SHOPIFY_API_SECRET` が設定され、それぞれのストアで認証が通ること |

手順の詳細は `docs/DEPLOY_PUBLIC_AND_INHOUSE.md` を参照。

## 4. 検証結果の記録方法（推奨）

- **チェックリスト**: 上記 3.2〜3.9 を表形式にし、実施日・結果（OK/NG）・備考を記入する。
- **不具合**: 再現手順・期待値・実際の挙動を短くメモし、Issue またはタスクに落とす。
- **管理画面UI**: `ADMIN_UI_DESIGN_RULES.md` および `ADMIN_UI_DIFF_POS_STOCK_VS_POS_RECEIPT.md` の項目を、検証時にあわせて確認すると、POS Stock との揃え漏れを防ぎやすい。

## 5. 今後の実装・品質まわりで検討するとよいこと

- **単体テスト**: API ルートやサービス（例: `settlementEngine`, `receiptApi`）の主要パスを vitest 等でテストし、CI で回す。
- **E2E**: 管理画面の「ログイン → 設定変更 → 保存」や、POS の「タイル → 検索 → 発行」までを Playwright/Cypress で自動化するかどうか検討。
- **管理画面UI**: DESIGN_RULES との差分を `ADMIN_UI_DIFF_POS_STOCK_VS_POS_RECEIPT.md` に従って順に解消する。
- **キャッシュ・ジョブ**: 売上サマリーの日次/期間キャッシュ更新の周期と、ジョブの有無を確認し、必要ならドキュメント化・実装する。
